# IntelliExam-Agent 产品与架构优化建议 (2026-03-27)

基于对当前项目架构（v3.1.0）、产品形态（三场景工作台、Planner-and-Solver 模式）以及教育命题/审题业务的理解，我梳理了以下几个维度的核心优化建议，旨在推动系统从“好用的辅助工具”向“真正的专家级智能协同体”演进：

---

## 💡 一、 交互与体验设计优化 (Interactive Design)

当前系统引入了 `Todo Board` (待办看板) 和 `Human-in-the-loop` 机制，这在可视化上非常现代，但在纵深体验上还可挖掘：

1. **局部重构与画布编辑 (局部 Agent 介入)**
   - **痛点**：目前“审核打回”或“用户意见重构”往往针对整道题或整个任务模块。
   - **优化**：在最终渲染的 Markdown 题卡中，提供类似 Notion AI 的“划词操作”。教师可光标选中某道题的某个选项或某段题干，触发局部指令（如：“仅将这个选项的错误设置得更隐蔽些”或“把这道题的背景换成航天科技”），系统仅针对局部调用 Solver 进行微调，而不是重跑整个大洋任务。
2. **Revision Diffs (版本对比流)**
   - **痛点**：当前系统支持最多3次的“命题-审核-修正”闭环，但用户很难直观看到 Agent“修正了什么”。
   - **优化**：在任务看板的结果区，针对经历过“被 Auditor 驳回”的题目，提供 `Diff 视图`（绿色新增、红色删除）。让教师能直观审阅 Agent 的修改轨迹，增加对 AI 能力的信任感。
3. **断点续查与会话快照**
   - **痛点**：复杂的命卷或审卷（如一套理综试卷）不仅耗时长，且用户可能需要中途挂起。
   - **优化**：完善 Todo 框架的“草稿/挂起”状态。支持不仅长效记忆（JSON）落地，也能将会话状态完整序列化，供用户“明天回来继续对这套卷子进行未完成的审阅”。

## 🎨 二、 产品功能与业务深度设计 (Product Design)

IntelliExam 需要更贴合真实的教学教研和出卷流程：

1. **结构化“试题银行”沉淀 (重构记忆落地)**
   - **痛点**：当前的长效记忆（Long-term memory）侧重于“偏好与经验”的文本总结，生成的优秀题目依然是非结构化的。
   - **优化**：新增一块“试题库（Question Bank）”模块。当试卷生成并经用户点赞/导出后，自动调用一个 `Extraction Agent`，将题目解析为极致结构化的 JSON（含题干、选项、解析、考点、难度标定），存入专用的题库数据库。在后续 Planner 阶段，可以直接复用或变体抽取。
2. **多维教学指标雷达图**
   - **优化**：在“审卷（Review）”和最终“命卷（Proposition）”的 Dashboard 中，不仅仅展示文本报告。可引入 `ECharts` 或 `Chart.js`，实时渲染一套卷子的多维指标指标：**知识点覆盖面分布图、布鲁姆层级（记忆/理解/.../创造）柱状图、难度梯度折线图**。让审题不仅是逻辑核查，更是卷面宏观质量的数据可视化。
3. **团队/教研组知识隔离**
   - **优化**：增加“个人记忆区”与“教研组公共知识基线”。出卷时往往需要遵循某个年级组统一的命题细则。将 RAG 知识库与本地 JSON 记忆切分为 `Global (School)`、`Team (Grade)` 和 `Personal` 三级，进行分层组合检索。

## 🧠 三、 Multi-Agent 系统与架构优化 (Architectural & Agent Design)

虽然 Planner-and-Solver 模式已经很先进，但在处理严谨的教育学科逻辑时，LLM 自身可能存在幻觉（尤其是理科），可以通过 Agent 编排来约束：

1. **引入“对抗审核模式” (Student Persona Agent)**
   - **痛点**：目前的 Auditor 依靠 prompt 来检查“逻辑矛盾”或“难度偏差”，对复杂的理科题目判定准确率难以保证。
   - **优化**：在 Review 环节，动态实例化一个或多个 `Student Agent`（例如：一个“中等生Agent”，一个“优等生Agent”），尝试去“做”这道题。
   - **逻辑**：如果 Student Agent 在做题时发现条件不足无法推导，或者得出多解，则直接反向证明题目存在科学性/规范性问题。这比单纯让大模型“看题挑错”要有效和精准得多。
2. **具象化学科专家池 (Domain-Specific Agents)**
   - **结合业务**：目前的 Solver 较为通用。可以重构为 `Solver Coordinator`，由它去调用具体的子 Agent，如：`Math_Expert_Agent`、`Physics_Expert_Agent` 等。
   - **硬性工具挂载**：为 `Math Expert` 强制挂载 `SymPy` 或 `Python REPL` (代码解释器) 工具，生成带有解析几何或复杂方程的题目时，强迫其写代码验证答案的唯一性和计算过程的准确性，根治理科命题的 LLM 算力软肋。
3. **长期记忆的自动“熵减”机制 (Memory Curator Agent)**
   - **痛点**：随着使用时间变长，`long_term_memory.json` 会堆积大量琐碎甚至矛盾的用户偏好（比如今天让出难题，明天让出简单题）。
   - **优化**：引入一个后台的延时/定时休眠期任务——`Curator Agent`（记忆结界师）。当系统空闲时，它负责读取全量 JSON 记忆，合并同类项，解决偏好冲突，降维总结，并打上时间权重衰减（Weight Decay）标签，保持记忆库的“少而精”。
4. **动态技能树 (MCP/Skills 异步加载图)**
   - **优化**：利用 MCP (Model Context Protocol)，不要在初始化时给所有 Agent 塞满工具集。Planner 在规划出 `TodoTask` 的同时，不仅指定下游 `Solver` 任务，同时**按需组装 Context 工具集**（类似给特种兵出发前配枪）。例如发现出题涉及“2026年最新时事政治”，才把 `web_search_tool` 和相应的 `news_mcp` 挂载进去，降低模型上下文噪音与 Token 消耗。

---

综上所述，后续的迭代可以围绕 **理科严谨度保障 (代码工具+对抗审核)、微观交互纵深 (局部渲染重试+Diff) 以及 数据赋能 (真正的试题库沉淀+可视化)** 来展开，这会极大提升产品的专业教研竞争护城河。
