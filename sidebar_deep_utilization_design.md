# 左侧边栏（任务看板）深度利用设计方案

## 1. 核心定位与设计愿景
目前左侧边栏在命题和审题模式下仅仅被当作“Agent 步骤日志（Activity Log）”使用，这浪费了 Multi-Agent 架构最大的优势——**“过程透明度”**与**“可干预性”**。

本设计方案旨在将左侧边栏的设计从**“单向的日志阅读器”**升级为**“双向的多智能体协作指挥台（Multi-Agent Orchestration Center）”**。让用户不仅能看见 AI 在干什么，还能随时打断、纠正或接管任何一个微小的命题步骤。

---

## 2. 核心交互链路（四大阶段）

我们以**“命制一份包含3道数学题的试卷”**为例，重新设计交互管线：

### Phase 1: 大纲共创与任务确认（Planning State）
* **动作**：用户在聊天栏发送“出3道高一函数题”。
* **后端**：`Planner Agent` 介入，不直接写题，而是输出一个 JSON 格式的“执行任务组（Todo Group）”。
* **左侧 UI**：渲染为一个可编辑的 Todo 看板。
  * 卡片 1：[待办] 命制题1 - 二次函数单调性 (选择题)
  * 卡片 2：[待办] 命制题2 - 指数函数图像 (填空题)
  * 卡片 3：[待办] 命制题3 - 对数函数应用 (解答题)
* **用户干预**：用户可以直接在左侧板上修改卡片2的描述为“对数函数综合大题”，或者删掉卡片3，最后点击底部的 **`[确认并开始生成]`** 按钮。

### Phase 2: 分布式执行与实时进度（Execution State）
* **动作**：点击确认后，`Manager` 将任务分发给不同的 `Solver Agent`。
* **左侧 UI**：任务卡片状态开始流转。
  * 卡片 1 显示 `[Running]`，并折叠展示内部步骤：“🔍 正在检索考纲库...”、“🔧 正在调用 MathEvaluator 验证答案...”。
  * 卡片 2 和 3 可能处于 `[Pending]` 排队中，或并发变成 `[Running]`。
* **右侧 UI**：右侧画布实时流式 (Stream) 浮现正在生成的题目文本。

### Phase 3: 对抗审核与驳回（Review & Debate State）
* **动作**：`Solver` 写完一题后，触发 `Review Agent` 进行质量审查或对抗审核。
* **左侧 UI**：
  * 如果 `Review Agent` 发现卡片 1 的选项 C 和 D 重复，卡片 1 状态变为橙色的 `[Need Revision]`。
  * 卡片下方自动展开一个像 GitHub 评论一样的 UI（Agent Comments），头像为 `[Reviewer]`，内容是：“选项设置不合理，缺乏区分度，请重试。”
  * 卡片 1 自动重置为 `[Running]` 开始根据建议自我修正。

### Phase 4: 局部接管与精修（Intervention State）
* **动作**：试题全部生成，右侧画布展示最终试卷。
* **用户干预**：如果用户不喜欢第 2 题，可以直接在左侧边栏找到“卡片 2”，点击卡片上的 **`[重新生成该题]`**（甚至可以补充一条自定义 Prompt：“换成更难的参数方程”）。
* **后端**：LangGraph 仅针对“题2”的节点进行重跑，右侧画布中的该题内容被无缝替换，**彻底消灭了以往需要重新生成整套试题的成本与不确定性**。

## 3. 前端 UI/UX 组件改造蓝图

在 `frontend/js/todo_board.js` 和 CSS 的基础上，我们需要增加以下组件特性：

1. **树形折叠面板 (Tree View)**: 
   - 一级：任务名称（如：生成第 1 题）
   - 二级：该任务的底层调用栈（如：Thought -> Tool Call: Retriever -> Reviewer Comment）
2. **交互性按钮组 (Action Toolbar)**: 
   - `[运行]` (Play)、 `[跳过]` (Skip)、 `[修改参数]` (Edit Config)、 `[溯源验证]` (Show Source)。
3. **Agent 角色标识 (Persona Avatar)**: 
   - 不同的操作显示是由哪个 Agent 完成的，比如 `[规划者 Planner]`、`[出题专家 Solver]`、`[审核员 Reviewer]`、`[工具 Tool]`，用不同颜色 Tag 或头像区分。

---

## 4. 后端架构的底层支撑需求

为支持上述交互，现有的 `LangGraph` 状态流需向以下方向改造：

1. **结构化指令输出**：优化 `agents/proposition/planner.py`，必须稳定输出符合特定 Schema 的 JSON/Pydantic 对象，而不是 Markdown 文本流。
2. **精细化 WebSocket 事件流**：
   原有：`{"type": "agent_step", "step": "生成题目"}`
   新增：`{"type": "todo_task_update", "task_id": "T001", "status": "running", "agent": "Math_Solver"}`
3. **LangGraph 的 Checkpoint 存取与局部重跑机制**：
   要实现由于某一道题不满意而进行的“局部重跑”，状态机中需要将生成的试题数组 (`state["draft_questions"]`) 进行实体化管理（每个 Item 有独立 ID，且关联到某个独立的 Graph 分支）。

---

## 5. 建议的分期落地路径

- **【一期】架构对接与静态渲染（高 ROI）**：
  将当前模型内部规划的“出题思路大纲”转为结构化数据，推给前端，由目前的 Todo 看板渲染出清晰的“解题步骤 / 出题步骤”，将黑盒变为白盒瀑布流。

- **【二期】卡片级错误捕捉与重试反馈（中等复杂）**：
  捕获 `MathEvaluator` 或 `Reviewer` 的错误信息，将其作为 Comment 挂载到左侧具体的任务卡片下方。让用户直观看到“AI 自己发现错误并改正”的过程闭环。

- **【三期】深度 Human-in-the-loop（高复杂）**：
  彻底打通用户对单张卡片的干预权（暂停/修改指令/单独重跑），实现真正的“指挥并接管流水线代理”体验。
