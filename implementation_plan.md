这是一份关于复杂AI Agent产品交互设计的深度产品思考文档，核心议题是如何将传统的"一问一答"Chat UI升级为类似Manus或Devin的Agent Workspace模式，以应对考试命题、审题等专业教育场景。

## 核心问题背景

传统Chat UI在处理复杂、长时、高专业度的Agent任务时存在明显缺陷：
- 用户产生"黑盒焦虑"：不知道AI在做什么、是否胡编乱造、还要等多久
- 缺乏过程透明度，导致信任度不足
- 在严肃教育领域，"证明过程"往往与结果同等重要

## 两大核心模式设计

### 一、命题命卷模式（Input: 指令 → Output: 文件）

**设计核心**：展示创造的推演过程，提供渐进式控制感

**交互形态：分屏工作台（Split-Screen Workspace）**
- 左侧/悬浮面板（Agent Activity Log / Thought Chain）：实时展示Agent内在思考、工具调用和执行步骤
- 右侧/主面板（Live Canvas / Preview）：实时预览最终产物生成过程（试卷大纲、逐步填充的试题）

**中间过程的结构化展示（Timeline/步骤卡片）：**
1. **意图识别与规划反馈**："收到需求，正在为您规划一套代数试卷（共30题，中等难度）。规划生成中..."
2. **知识与标准检索**：展示动态工具调用状态，如"[正在搜索网络] 最新数学课程标准..."、"[查阅知识库] 高中代数常考题型分布..."，可做成折叠面板展开查看具体检索Snippet
3. **试卷蓝图设计（关键Human-in-the-loop节点）**：大规模生成题目前，Agent先产出"组卷大纲/双向细目表"（知识点分布、题型、配分），暂停等待用户"确认生成"或"调整大纲"，避免长时任务跑错方向
4. **题目并发生成**：进度条或矩阵（如30个小方块）显示每题生成状态（生成中→自检中→完成）
5. **质量审核与排版**："[执行排版] 正在转换为标准Word格式..."

**交互亮点：可干预性（Steerability）**
- 生成过程中允许用户随时"中止并修正"
- 示例：用户发现Agent检索了错误教材版本，可立刻打断补充指令："请使用人教版教材"

### 二、审题审卷模式（Input: 文件/长文本 → Output: 质检报告）

**设计核心**：证据的可追溯性，沉浸式的双流对比

**交互形态：文档批注视图（Document Audit View）**
- 类似代码审查或现代文档协作工具
- 主区域（原稿视图）：渲染上传的试卷内容，Agent发现问题时直接在对应题目上高亮/画线
- 侧边栏（审查Timeline & 报告单）：动态生成审查流，最终汇总为可交互报告

**中间过程的结构化展示：**
1. **文件解析与指纹识别**："[文件解析中] 正在提取试卷文本与公式..." → "[识别完毕] 检测到物理试卷，包含15道选择题，5道大题"
2. **规则加载与爬网**：展示审题依据，"[加载审题规则] 正在获取《2024物理中考命题规范》..."，"[搜索] 正在全网检索是否有原题/查重..."
3. **逐题扫描与推理（高光时刻）**：
   - 视觉上题目被一道道"扫过"（扫描线或高亮状态切换）
   - 侧边栏实时弹出Agent思考卡片："正在验证第3题的受力分析...发现矛盾！选项C中的摩擦力计算未考虑静摩擦临界点"
   - 查重提示："警告：第5题在2023年北京卷中出现过，匹配度95%（附上来源链接）"
4. **综合报告生成**：最终报告不仅是可下载PDF，更是当前页面上的结构化Dashboard（科学性错误X处、格式错误Y处、难度超标Z处等），用户点击报告项可联动滚动到原试卷对应位置

## 技术实现建议（类Manus体验）

### 协议层分离
- 后端Agent返回的不是纯Markdown文本，而是具有类型的事件流（Event Stream）
- 事件类型：ThoughtEvent、ActionEvent（调用搜索）、ObservationEvent（获取网页内容）、OutputEvent（局部生成结果）

### 状态指示器（Micro-interactions）
- 大量使用微动画表现"正在思考"、"正在爬取"、"正在计算"
- 让长时任务的等待过程变得"好看且让人觉得它很努力"

### 保留对话框的兜底作用
- 即使进入工作台模式，底部或侧边保留迷你控制台/对话框
- 用户可直接输入局部修正指令（如"第3题太难了，换一道"）
- Agent直接在工作台视图上进行局部重绘（Local Update），而非重新走完整流程

## 产品架构：基础对话 + 进阶模式切换

### 架构理念
- **渐进式展开（Progressive Disclosure）**与**情境化工作区（Contextual Workspace）**结合
- 用户绝大多数时候只需轻量级对话，触发专业需求时系统"变形"为重实操的生产力工具

### UX/UI切换体验

**意图拦截与平滑切入**
- 用户手动点击"新建命题任务"、"新建审卷任务"按钮
- 系统底层具备敏锐"意图识别"能力：当用户在对话框输入明确命题诉求（如"帮我出一套涵盖微积分基础的期末冲刺卷，带详细解析"），对话模型（LLM Router）识别为【命卷意图】
- 在聊天流中渲染"任务确认卡片（Task Card）"，显示提取参数（类型=命卷，题材=微积分，难度=冲刺，带解析）
- 提供醒目按钮："🚀 开启独立命卷空间"或"确认生成"
- 视觉过渡：点击确认后，聊天视图平滑缩放或侧边滑出（Sliding Panel），变为分屏工作台模式，原聊天框变为工作台底部悬浮输入框

**状态的隔离与保持**
- 对话模式：快速、碎片化的信息交换，"无状态"或"轻状态"
- 进阶模式（工作台）："有状态（Stateful）"的工程项目，每个任务对应唯一TaskID或SessionID，支持断点续作

### 技术架构支撑

**1. 统一网关与意图路由（Manager Agent / Router）**
- 用户所有输入先经过"大堂经理（Manager Agent）"
- 日常问答（如"牛顿第一定律是什么？"）：走常规RAG或LLM回复
- 结构化高维任务（上传物理卷子、完整命题诉求）：Manager Agent提取意图，转发给专门"命卷Sub-Agent"或"审卷Sub-Agent"，向前端发送模式切换信号（SwitchModeEvent）

**2. 统一事件流通信（Unified Semantic Event Stream, SSE/WebSocket）**
- 进阶模式任务可能长达几分钟至十几分钟，必须使用双向实时通信
- 前端消化不同类型事件：
  - `status_update`：通知界面更新进度和状态指示灯
  - `tool_call & tool_result`：告知当前执行动作，展示Agent思考链
  - `content_delta`：实时流式传输工作台右侧画布预览内容
  - `interrupt_request`：Agent遇到不确定条件时主动挂起，向前端发起询问卡片

**3. 工作流持久化（Task Persistence）**
- 复杂长时任务易中途出错（断网、API频率限制）
- 后端必须包含坚实的状态机框架（如LangGraph或基于数据库的执行上下文日志）
- 确保每道题生成状态、每步审卷结果都能落库，失败时可在该节点重试，而非让用户从头等待

## 结论与开发主线

**产品价值**：这种设计是将普通"套壳AI"拉开身位、"做厚"产品壁垒的关键——保留基础对话保障零学习成本和广泛适用场景；引入进阶工作台真正触及B端和严肃教育场景痛点，提供专业感和控制感。

**并行开发主线**：
1. **产品与前端侧**：设计实现基础对话界面、卡片化交互组件，以及能够渲染丰富事件流的分屏"工作台（Workspace）" UI引擎
2. **底层引擎侧**：强化Multi-Agent路由能力，统一事件流通讯协议（定义严谨通信格式驱动前端不同UI状态切换），完善核心专业Agent（命题Agent、审阅Agent、检索Agent）的健壮执行能力
